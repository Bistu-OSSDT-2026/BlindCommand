# 贡献指南 CONTRIBUTING.md
欢迎为本项目贡献代码、文档与测试！本项目采用 MIT 开源协议，所有提交行为均遵循本规范。

## 一、可参与的贡献类型
1. 新功能开发（对应 feature Issue）
2. Bug 缺陷修复（对应 fix Issue）
3. 文档优化：README、使用说明、开发文档
4. 自动化配置：GitHub Actions 测试、代码检查
5. 项目结构、代码格式优化

## 二、标准完整贡献流程（必须严格遵守）
1. 查看仓库 Issue 列表，选择未被认领的任务；无对应任务则新建 Issue，写清任务内容、验收标准、负责人
2. 认领 Issue，创建独立开发分支，禁止直接在 main 主分支修改代码
3. 本地完成编码，自测确保功能可正常运行
4. 规范提交代码，推送远程分支，提交 Pull Request（PR）
5. 其他组员交叉评审 PR，提出修改意见
6. 根据评审意见修改代码，更新 PR
7. 评审全部通过后，项目负责人合并 PR 至 main 分支
8. 若PR关联Bug/功能Issue，合并后自动关闭对应Issue

## 三、分支命名规范
所有分支必须使用统一前缀，格式：`前缀/issue编号-简短描述`
- feature/：新增功能，例：`feature/#3-md-catalog`
- fix/：Bug修复，例：`fix/#5-file-read-error`
- docs/：文档修改，例：`docs/#7-update-readme`
- chore/：配置、自动化、打包等杂项工作

## 四、Git提交注释规范
统一格式：`类型: 简短描述`
类型可选：
- feat：新增功能
- fix：修复bug
- docs：文档更新
- chore：配置/工具修改
示例：
feat: 实现MD文件解析标题功能
fix: 修复文件空内容报错问题

## 五、Pull Request（PR）提交要求
1. PR标题清晰，写明本次修改内容
2. PR描述中必须关联对应Issue，语法：Fixes #编号（合并后自动关闭Issue）
3. 自检清单提交前全部完成：
- [ ] 本地运行无报错
- [ ] 完成对应Issue全部需求
- [ ] 通过GitHub Actions自动化检查
- [ ] 无多余无关代码、注释

## 六、代码评审（Review）规则
1. 禁止自己审核自己提交的PR，必须交叉互审
2. 评审重点检查三点：
   1）代码是否完整实现Issue需求
   2）本地拉取分支可正常运行
   3）代码风格、目录结构符合项目设计约定
3. 存在问题在线留言，开发者修改后重新推送更新PR，再次复核通过才可合并

## 七、Bug 处理流程
1. 发现异常 → 新建Bug类型Issue，写明复现步骤、报错信息、预期效果
2. 组员认领Bug Issue，新建fix分支修复代码
3. 提交PR，评审验证修复效果
4. 合并代码，关闭Bug Issue

## 八、版本发布规范
最终发布版本固定为 v0.1.0，Release Notes 写明版本功能、使用方法、已知缺陷。
